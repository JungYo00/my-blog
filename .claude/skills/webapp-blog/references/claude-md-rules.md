# CLAUDE.md 규칙 원문

이 프로젝트(`my-blog`)의 `/CLAUDE.md`에 실제로 적용 중인 규칙 전문이다. 이 스킬이 트리거되면 이 규칙들이 곧 작업 절차이므로, 실행 전에 프로젝트 루트의 `CLAUDE.md`를 직접 읽어 최신 상태인지 확인할 것 — 이 파일은 스킬을 만든 시점의 스냅샷이라 이후 사용자가 규칙을 바꿨을 수 있다.

```markdown
## 작업 사이클
1. Plan — 서브에이전트를 만들어 spec.md를 작성시킨다. 사용자 승인을 받는다.
2. Work — 서브에이전트를 만들어 구현한다. 화면이 3개 이상이면 화면별로 서브에이전트를 나눈다.
3. Review — 서브에이전트를 만들어 테스트하고 review.md를 작성시킨다.
4. Commit — 기능마다 깃 커밋하고 깃허브에 푸시한다.

## 서브에이전트 규칙
- 서브에이전트에게 작업을 넘길 때 전용 지침 파일(.md)을 만들어 전달한다.
- 서브에이전트는 지침 파일에 명시된 범위만 수정한다.
- Work가 복잡하면 여러 서브에이전트로 나눈다. 각각 독립적으로 작업할 수 있게 범위를 겹치지 않게 나눈다.

## 웹앱 규칙
- 모든 웹앱은 블로그의 색상 팔레트를 따른다. style.css의 CSS 변수를 참조할 것.
- 웹앱에 사용법 안내 문구를 반드시 포함한다.

## 규칙
- 승인 없이 구현을 시작하지 않는다.
- 막히면 사용자에게 알린다.
```

## 왜 이 순서인가

- **Plan이 먼저인 이유**: spec.md 없이 바로 구현하면 사용자가 승인/거부할 대상이 없다. 특히 이 프로젝트는 매 기능마다 사용자가 직접 확인하고 싶어 하는 세부 결정(예: 헤더 링크 추가 여부, 저장 파일명 고정 여부, 모바일 입력 방식)이 있어서, 그런 결정들을 spec.md의 "결정 사항" 절에 명시적으로 남기고 승인받는 절차가 실제로 반복 재사용되고 있다.
- **Work를 화면 수로 나누는 이유**: 화면이 여러 개면 한 서브에이전트가 맥락을 다 들고 있기보다, 화면별로 범위를 쪼개 병렬로 작업시키는 편이 빠르고 충돌이 적다. 단, 파일 범위가 겹치면 안 되므로 지침 파일에 "이 파일만 수정" 목록을 명시해야 한다.
- **Review가 절대 코드를 고치지 않는 이유**: Review 서브에이전트가 발견한 버그를 스스로 고치기 시작하면 review.md가 "고치기 전 상태"를 정확히 기록하지 못하게 된다. 실제로 pixel-art-editor 기능에서 Review가 발견한 버그(games/2048 헤더에 새 네비 링크 누락)는 메인 대화(오케스트레이터)가 review.md에 기록된 내용을 보고 직접 고쳤다 — Review 서브에이전트 자신이 고치지 않았다.
- **기능마다 커밋하는 이유**: 여러 기능을 한 커밋에 몰아넣으면 나중에 특정 기능만 되돌리거나 리뷰하기 어렵다.
- **웹앱 규칙(색상 팔레트, 사용법 안내)이 별도로 있는 이유**: 이 프로젝트의 웹앱(2048, 픽셀 아트 등)은 블로그 본문과 별도의 정적 페이지지만 동일 사이트의 일부로 느껴져야 한다. 색을 하드코딩하면 다크모드 전환 시 그 웹앱만 안 바뀌는 회귀가 생기고, 사용법 문구가 없으면 처음 방문한 사용자가 조작법을 모른다. 실제로 2048은 "방향키 또는 스와이프로 타일을 밀어 합쳐보세요", 픽셀 아트는 "격자를 클릭하거나 드래그해서 도트를 찍어보세요" 문구를 화면 상단에 두고 있다.

## 새 프로젝트에 이 규칙을 이식할 때

이 스킬을 다른 블로그+웹앱 프로젝트에 적용하는 경우, 대상 프로젝트에 `CLAUDE.md`가 없다면 위 내용을 그대로 새 `CLAUDE.md`로 만들어 제안하되, "웹앱 규칙"의 색상 팔레트 언급은 그 프로젝트의 실제 공통 스타일시트 경로에 맞게 바꿔야 한다(예: `style.css`가 아니라 `styles/theme.css`일 수 있음).
