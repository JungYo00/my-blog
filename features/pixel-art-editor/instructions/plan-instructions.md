## 기존 프로젝트 컨텍스트 (반드시 읽고 반영할 것)
- 이 저장소는 마크다운 → 정적 블로그 사이트를 만드는 SSG 프로젝트다. 프레임워크/번들러/npm 의존성 없이 순수 HTML/CSS/JS + Node 내장 모듈로만 되어 있다.
- `build.js`를 반드시 읽어라. 이전에 "2048 게임" 기능을 추가하면서 이미 한 번 이 패턴을 썼다: 저장소 최상위에 `games/2048/` 소스 디렉토리를 두고, `build.js`에 `const GAMES_SRC = path.join(ROOT, 'games', '2048');`를 추가한 뒤 `fs.existsSync(GAMES_SRC)`일 때만 `fs.cpSync(GAMES_SRC, path.join(DIST_DIR, '2048'), {recursive:true})`로 `dist/2048/`에 복사하도록 만들어 두었다. 이번 픽셀 아트 에디터도 **같은 패턴**을 따른다: 저장소 최상위에 새 소스 디렉토리(예: `pixel-art/`)를 만들고, build.js에 동일한 방식으로 `PIXEL_ART_SRC` 상수 + 조건부 복사를 추가해 `dist/pixel-art/`로 배포되게 한다. (게임이 아니므로 `games/` 하위에 넣지 말고 저장소 루트에 독립 디렉토리로 둘 것을 제안 — 네가 spec에서 최종 확정)
- `templates/layout.html`을 읽어라. 헤더(`.site-header`)에 `site-title` 링크, "2048" 내비게이션 링크(`.nav-link`), 다크모드 토글 버튼(`#theme-toggle`)이 이미 있다. 이번에도 헤더에 내비게이션 링크를 추가할지 spec에서 다뤄라.
- `css/style.css`를 읽어라. `--color-bg`, `--color-bg-secondary`, `--color-text`, `--color-text-secondary`, `--color-border`, `--color-link`, `--font-sans`, `--font-mono` 등 다크모드 CSS 커스텀 프로퍼티가 이미 정의돼 있다. 새 기능도 이 토큰을 재사용해서 다크모드에 자동 대응하게 해야 한다.
- `games/2048/index.html`을 읽어라 — 독립 정적 페이지가 `templates/layout.html`의 다크모드 토글/FOUC 방지 인라인 스크립트 패턴을 어떻게 복제했는지 참고용 예시로 확인해라. 이번 기능도 같은 패턴을 따를지 spec에서 판단해라.
- 외부 라이브러리 없음 원칙 유지: PNG 저장은 브라우저 내장 `<canvas>`의 `canvas.toBlob()` / `canvas.toDataURL('image/png')` API로 구현 가능하니 별도 라이브러리가 필요 없다는 점을 spec에 명시해라.

## spec.md에 반드시 포함할 내용
1. **개요**: 기능 한 줄 요약, 배치 위치(URL 경로 제안), 블로그 헤더 링크 추가 여부(제안하고 "결정 필요"로 표시)
2. **에디터 기능 상세**:
   - 16x16 격자 캔버스. 격자 한 칸(도트)당 크기(예: 화면에서 24~32px 정도로 확대해서 그리기 쉽게) 제안
   - 마우스 클릭으로 도트 찍기, **드래그(마우스를 누른 채 이동)로 연속으로 칠하기** 지원 여부 — 표준 픽셀 에디터 UX이므로 포함 제안하고 "결정 필요"로 표시
   - 지우개 기능(우클릭 또는 별도 지우개 버튼으로 투명/배경색 처리) 포함 여부 — 제안
   - 전체 지우기(Clear) 버튼
   - 격자를 canvas 하나로 그릴지, 16x16 개별 DOM 엘리먼트(div)로 그릴지 기술적 접근 방식을 비교하고 하나를 제안 (힌트: canvas가 PNG 저장에 직접적으로 유리함 — `canvas.toDataURL('image/png')`으로 바로 저장 가능. DOM div 방식은 저장 시 별도로 canvas에 다시 그려야 함. 이 프로젝트 관례상 단순함을 우선하므로 canvas 기반을 권장하되 최종 판단은 네가 spec에 명시)
3. **색상 팔레트**:
   - 기본 제공 팔레트 색상 개수/구성 제안 (예: 16색 정도의 기본 팔레트 - 흑백, 원색, 파스텔 등)
   - 커스텀 색상 선택(HTML `<input type="color">`) 추가 여부 — 제안
   - 현재 선택된 색상을 시각적으로 표시(테두리 강조 등)
4. **PNG 저장**:
   - "저장" 버튼 클릭 시 `canvas.toDataURL('image/png')` 또는 `canvas.toBlob()` + `<a download>` 클릭 트리거로 파일 다운로드
   - 저장 시 실제 해상도 제안: 16x16 원본 그대로 저장할지(작은 이미지), 화면 표시 배율(예: 16배 확대해 256x256으로 저장)로 업스케일해서 저장할지 — **결정 필요**로 표시하되 업스케일 저장을 기본값으로 제안 (16x16 그대로는 실사용성이 낮음). 업스케일 시 이미지가 흐려지지 않도록 `imageSmoothingEnabled = false` 적용을 명시.
   - 파일명 제안 (예: `pixel-art.png` 또는 타임스탬프 포함)
5. **화면/컴포넌트 구성**: 화면 개수 판단 (CLAUDE.md 규칙: 3개 이상이면 화면별 Work 서브에이전트 분리 — 이 기능은 아마 1개 화면으로 충분할 것으로 예상, 네 판단을 적어라)
6. **파일 구조 제안**: 새로 만들 파일 목록과 책임, build.js에서 수정할 부분(정확히 어떤 줄 근처에 추가할지)
7. **접근성/반응형**: 모바일에서 터치로 도트 찍기 지원 여부(터치 이벤트도 마우스 드래그처럼 동작해야 자연스러움 — 제안), 좁은 화면에서 격자+팔레트 레이아웃이 어떻게 배치될지
8. **테스트 관점**: Review 단계에서 검증할 시나리오 목록 초안 (클릭으로 도트 찍기, 드래그로 연속 칠하기, 색상 변경, 지우개, 전체 지우기, PNG 저장 후 다운로드된 파일 확인, 모바일 터치, 다크모드, 기존 블로그/2048 게임 회귀 없음)
9. **미해결 질문**: 네가 판단하기 애매하거나 사용자 확인이 필요한 항목 명시적으로 나열
