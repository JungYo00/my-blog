# 2048 게임 Review

## 요약
`node build.js` 실행 후 `dist/2048/`가 정상 생성되고, `node serve.js`(포트 3000)로 서빙된 실제 페이지에서 spec.md 7절의 13개 검증 시나리오를 모두 실제 브라우저 조작(키보드 이벤트, 클릭, localStorage 조회, DOM 판독)으로 확인했다. **13개 시나리오 모두 PASS**이며, 게임 로직(이동/병합/재병합 방지/게임오버/승리/점수/BEST 저장/다크모드/헤더 연동) 자체에서 버그는 발견되지 않았다. 다만 테스트 중 발견한 낮은 심각도의 관찰 사항 1건(모바일 방향 버튼에 대한 자동화 클릭 도구의 히트테스트 불안정 — 실제 게임 코드 문제는 아님으로 판단)을 아래에 기록한다.

## 발견된 버그 (있는 경우)
게임 로직/코드 자체에서 확인된 버그는 없음.

### [관찰 사항 / 심각도: Low] 모바일 방향 버튼에서 `preview_click` 도구의 히트테스트 불안정 (코드 버그로 확신할 수 없음)
- 재현 방법: `preview_resize(mobile)` 후 `.touch-left`, `.touch-right` 버튼에 대해 `preview_click` 도구로 클릭 시도
- 기대 동작: 클릭 시 해당 방향으로 타일 이동
- 실제 동작: `preview_click`으로는 이동이 전혀 발생하지 않음(타일 상태 완전 동일). 그러나 같은 버튼에 대해 (a) `el.click()` JS 직접 호출, (b) `pointerdown/mousedown/pointerup/mouseup/click` 합성 이벤트 시퀀스를 `dispatchEvent`로 직접 발생시킨 경우 모두 정상적으로 이동이 발생함을 확인. `elementFromPoint`로 버튼 중심 좌표에 다른 엘리먼트가 겹쳐있지 않음도 확인(정상적으로 버튼 자신이 최상단)
- 관련 파일: `games/2048/index.html`(`.touch-controls` 버튼), `games/2048/game.js`(`touchButtons.forEach(...addEventListener('click', ...))`) — 코드 자체는 표준 `click` 리스너를 사용하고 있어 실제 사용자 클릭/탭에는 정상 반응할 것으로 판단됨. 이번 이슈는 자동화 테스트 도구(`preview_click`)가 작은 크기(52x52px, flex 배치)의 버튼에 대해 좌표를 다소 다르게 계산하는 것으로 보이는 테스트 인프라 측 현상이며, 게임 코드의 결함으로 보기 어려움. 실사용자 환경에서 재확인이 필요하면 실제 모바일 기기 테스트를 권장

## 스크린샷/증거
- 초기 로드(다크모드): 4x4 보드에 타일 2개(값 2), SCORE/BEST 0/0, 새 게임 버튼, 방향 버튼 4개 모두 확인
- 게임 오버 스크린샷: 보드가 가득 찬 상태(4,8,4,2 / 2,64,16,64 / 16,4,128,2 / 2,256,2,16 — 인접 동일값 없음)에서 "Game Over" 오버레이 + "다시하기" 버튼이 정확히 표시됨을 확인
- 승리(2048) 상태: 결정론적 시퀀스 재생 후 `maxVal:2048, win:true, score:19496`을 `preview_eval`로 직접 확인. 이어서 "계속하기"(오버레이만 닫히고 게임 지속) / "새 게임"(전체 초기화) 버튼 둘 다 동작 확인
- 모바일(375x812) 스크린샷: 헤더, 점수판, 새 게임 버튼, 4x4 보드, 방향 버튼 4개가 375px 폭 안에 모두 들어오고 가로 스크롤이 없음을 확인
- 라이트모드 스크린샷: 다크모드 토글 클릭 후 헤더/점수판/보드/타일/방향버튼 전체가 밝은 배색(연한 배경 + 어두운 텍스트, 타일은 베이지 톤)으로 자연스럽게 전환됨을 확인
- 네트워크: `2048/index.html`, `2048/style.css`, `2048/game.js`, `css/style.css`, `js/theme.js` 모두 200 OK로 정상 로드. 실패한 요청은 기존 포스트의 외부 placeholder 이미지 1건뿐(샌드박스 네트워크 제약, 2048 기능과 무관)
- 참고: 승리 시퀀스 재생 도중 한 차례 SCORE가 0으로, 보드가 초기 2타일 상태로 보인 순간이 있었으나, 이는 BEST 값이 정확히 localStorage에 반영된 상태(`best:19496`)와 URL 불변으로 미루어 볼 때 페이지 리로드(자동 새로고침)에 의한 것으로 추정되며, 이후 반복 검증한 새 게임/다시하기/계속하기 버튼은 모두 클릭한 시점에만 정확히 예상대로 동작해 게임 코드 자체의 의도치 않은 리셋 버그로는 보이지 않음
