# 2048 게임 기능 spec

## 1. 개요
- **한 줄 요약**: 방향키로 숫자 타일을 밀어 합치는 표준 2048 퍼즐 게임을 블로그에 정적 페이지로 추가한다.
- **배치 위치(제안)**: `/2048/` (즉 `dist/2048/index.html`). 저장소 최상위 소스 디렉토리는 `games/2048/`로 제안 (build.js가 이를 `dist/2048/`로 복사).
- **블로그 메인 페이지와의 연결 (결정 필요)**: `templates/layout.html`의 `.site-header`에 "2048" 링크를 추가할지, 아니면 완전히 독립된 숨은 경로로만 둘지 결정 필요. 제안: 헤더의 site-title 옆에 텍스트 링크(`<a href="{BASE}2048/index.html">2048</a>`) 추가. 단, 이 경우 `templates/layout.html` 및 그 결과로 매 포스트/인덱스 페이지가 변경되므로 Work 단계에서 영향 범위가 넓어짐. 링크 추가 없이 URL을 아는 사람만 접근하는 방식도 가능.

## 2. 게임 규칙 (표준 2048 규칙)
- 4x4 격자(16칸).
- 새 게임 시작 시 빈 칸 중 무작위 2곳에 타일 생성. 값은 90% 확률로 2, 10% 확률로 4 (표준 관례).
- 방향키 ↑↓←→ 입력 시 모든 타일을 해당 방향의 끝까지 이동시키며, 이동 경로상 같은 값의 타일이 부딪히면 합쳐져 값이 2배가 되고 점수에 합쳐진 값만큼 가산.
- 한 번의 이동(키 입력 1회) 동안 이미 합쳐진 타일은 같은 이동 내에서 다시 합쳐지지 않는다 (예: `2 2 4` → 오른쪽 이동 시 `_ 2 8`이 되어야지 `_ _ 16`이 되면 안 됨. 병합된 타일에 플래그를 두고 같은 턴에는 재병합 금지).
- 이동 결과 격자에 실제 변화(위치 이동 또는 병합)가 있었을 때만 새 타일 1개를 빈 칸에 랜덤 생성. 변화가 없으면(막힌 벽쪽으로 이동 시도 등) 아무 일도 일어나지 않음.
- 2048 값의 타일이 생성되면 승리 처리. **계속하기 옵션 포함 여부는 결정 필요** (아래 미해결 질문 참고). 제안: 승리 오버레이에 "계속하기" / "새 게임" 두 버튼 제공.
- 더 이상 유효한 이동(모든 칸이 채워져 있고 인접한 동일 값 쌍도 없음)이 불가능하면 게임 오버 처리.
- 새 게임(재시작) 버튼 상시 노출.

## 3. 점수판 UI
- 게임 보드 상단에 두 개의 박스: "SCORE"(현재 점수), "BEST"(최고 기록).
- 최고 기록은 `localStorage`(예: 키 `game2048.bestScore`)에 영속 저장, 새로고침/재방문 후에도 유지.
- 현재 점수가 최고 기록을 넘으면 BEST 값을 그 자리에서 갱신 후 저장.
- 스타일: 기존 `css/style.css`의 다크모드 커스텀 프로퍼티(`--color-bg`, `--color-bg-secondary`, `--color-text`, `--color-text-secondary`, `--color-border` 등)를 그대로 활용해 점수 박스를 `--color-bg-secondary` 배경 + `--color-border` 테두리의 카드 형태로 구성. `[data-theme]` 전환에 자동으로 대응되도록 하드코딩 색상 금지.

## 4. 화면/컴포넌트 구성
이 기능은 **화면 1개**로 판단한다 (CLAUDE.md 규칙: 화면 3개 이상이면 화면별 Work 서브에이전트 분리 필요 — 해당 없음).
- 메인 게임 화면(보드 + 점수판 + 새 게임 버튼) 하나만 존재.
- "게임 오버"와 "승리(2048 달성)"는 별도 라우트/페이지가 아니라 같은 화면 위에 뜨는 **오버레이(모달)** 컴포넌트로 처리 (기존 페이지 위에 반투명 배경 + 메시지 + 버튼). 별도 화면으로 세지 않음.
- 따라서 Work 단계는 단일 서브에이전트로 진행 가능하다고 제안.

## 5. 파일 구조 제안
저장소 최상위에 신규 소스 디렉토리 `games/2048/`를 만들고, build.js가 `dist/2048/`로 복사한다.

```
games/2048/
├── index.html      # 게임 페이지 마크업 (보드, 점수판, 버튼, 오버레이). 자체 <html> 문서 (templates/layout.html 시스템을 타지 않는 독립 정적 페이지)
├── style.css       # 게임 전용 스타일 (그리드, 타일, 오버레이). css/style.css의 커스텀 프로퍼티를 재사용하기 위해 index.html에서 ../css/style.css와 함께 로드
└── game.js         # 게임 로직: 상태(4x4 배열), 이동/병합 알고리즘, 렌더링, 키보드 입력 처리, localStorage 최고 기록, 새 게임/승리/게임오버 처리
```

**build.js 수정 필요 사항**:
- 상단 상수에 `const GAMES_SRC = path.join(ROOT, 'games', '2048');` 추가.
- `main()` 함수 내, 기존 `fs.cpSync(CSS_SRC, ...)` / `fs.cpSync(JS_SRC, ...)` 복사 로직 바로 아래에 `fs.cpSync(GAMES_SRC, path.join(DIST_DIR, '2048'), { recursive: true });` 추가.
- `games/` 디렉토리 자체가 없거나 비어있을 때 빌드가 죽지 않도록 존재 여부 체크(`fs.existsSync`) 후 조건부 복사 권장.
- (헤더 링크를 추가하기로 결정하면) `templates/layout.html`의 `.site-header` 블록에 링크 1줄 추가.

`index.html`은 `css/style.css`를 상대경로(`../css/style.css`)로 참조하고, `js/theme.js`도 다크모드 토글 버튼을 넣을 경우 함께 로드(`../js/theme.js`)한다. 다크모드 토글 버튼 포함 여부도 결정 필요(아래 참고) — 포함 시 `templates/layout.html`과 동일한 `#theme-toggle` 버튼 + 초기 테마 적용 인라인 스크립트를 game/index.html에도 복제해야 함.

## 6. 키보드/접근성
- 방향키(↑↓←→) `keydown` 이벤트에서 `event.preventDefault()` 호출해 페이지 스크롤 방지.
- 포커스가 입력 요소(있다면)에 있을 때는 방향키를 게임에 반응시키지 않도록 예외 처리 고려(이 게임엔 텍스트 입력 요소가 없을 것으로 예상되어 우선순위 낮음).
- 터치/모바일 스와이프 지원 (**선택 사항**): 모바일에서는 방향키가 없으므로 `touchstart`/`touchend` 좌표 차이로 스와이프 방향을 판별해 이동 트리거. 최소 구현으로는 화면 하단에 방향 버튼(↑↓←→) 4개를 두는 대안도 가능. 둘 중 하나는 필요 — 순수 방향키만 지원 시 모바일에서 플레이 불가.

## 7. 테스트 관점 (Review 단계 검증 시나리오 초안)
- 정상 이동: 4방향 각각에서 타일이 끝까지 밀리는지.
- 정상 병합: 같은 값 2개가 만나 합쳐지고 점수가 가산되는지. `2 2 2 2` → 오른쪽 이동 시 `_ _ 4 4`가 되는지(같은 턴 재병합 금지 검증).
- 이동 후 변화 없을 시 새 타일이 생성되지 않는지 (예: 이미 벽에 붙어있는 상태에서 그 방향으로 다시 이동 시도).
- 게임 오버 판정: 보드가 가득 차고 인접 동일값이 전혀 없을 때 정확히 게임 오버로 전환되는지.
- 승리 판정: 2048 타일 생성 시점에 승리 오버레이가 뜨는지, "계속하기" 선택 시 게임이 계속되는지.
- 점수 계산: 여러 번의 병합이 한 이동에 걸쳐 발생할 때 점수 누적이 정확한지.
- 새로고침 후 최고 기록 유지: localStorage에 저장된 BEST 값이 페이지 리로드 후에도 표시되는지, 새 게임에서 SCORE가 BEST를 넘었을 때만 갱신되는지.
- 모바일 반응형: 좁은 화면(375px 등)에서 보드/타일 크기가 화면을 벗어나지 않는지, 스와이프(또는 방향 버튼)가 동작하는지.
- 다크모드: `[data-theme="dark"]`/`prefers-color-scheme: dark`에서 보드/타일/점수판 색상이 커스텀 프로퍼티를 따라 자연스럽게 전환되는지, 라이트모드 대비 가독성.
- 빌드 검증: `node build.js` 실행 후 `dist/2048/index.html`, `dist/2048/style.css`, `dist/2048/game.js`가 정상 생성되는지, 기존 포스트/인덱스 빌드에 영향이 없는지.

## 8. 결정 사항 (사용자 확정, 2026-07-10)
1. **게임 오버 후**: 오버레이 + "다시하기" 버튼으로 리셋 (기본안 그대로 채택).
2. **승리(2048 도달) 후**: "계속하기" 옵션 포함. 오버레이에 "계속하기"/"새 게임" 버튼 모두 제공, 계속하기 선택 시 4096/8192 등으로 이어서 플레이 가능.
3. **블로그 헤더 링크**: 노출한다. `templates/layout.html`의 `.site-header`에 "2048" 링크 추가 (모든 포스트/인덱스 페이지에 반영됨).
4. **게임 페이지 다크모드 토글**: `templates/layout.html`과 동일하게 `#theme-toggle` 버튼 포함 (일관된 UX).
5. **모바일 입력**: 스와이프 제스처 + 화면 방향 버튼 둘 다 지원.
6. **애니메이션**: CSS transition으로 타일 이동을 부드럽게 처리.
7. **디렉토리/URL 네이밍**: `games/2048/` 소스 → `/2048/` 경로. 이견 없음, 확정.
8. **페이지 메타**: `<title>2048 게임 | My Blog</title>`, description "방향키로 숫자 타일을 밀어 합치는 2048 퍼즐 게임".
